Project Anvil Guidelines
Code of Conduct and Integrity
Violation of any of the following rules will result in immediate removal from Project Anvil.
1. Originality of Code
* Private Repository: Your repository must remain private.
* No Prior Publication: Your code must never have been published online previously.
* No Reuse: Do not reuse code from public repositories.
2. Task Independence
* Each task must start from the same clean base repository.
* Tasks must not depend on changes or states from other tasks.
________________


Project Specifications
You are required to create:
* 1 Private Repository: This will serve as your project base.
* 10 Independent Tasks: Each task must test a realistic engineering change.
Task Components
Each task must include:
* Problem Description: A clear statement of the issue or feature request.
* Tests: A suite of tests defining correct behavior.
* Gold Solution: A correct solution patch.
* Metadata: Essential data for evaluation.

Quality Standards
A task is considered high quality if it meets the following criteria:
1. Realistic Engineering Scenarios
* Tasks should simulate work performed by a professional developer.
* Avoid trivial refactors or "toy" functions.
2. Multi-File Changes
* Solutions must require changes across at least 3 files.
   * Example: Interface updates + Implementation + Routing logic.
   * Example: Configuration + Validation + Documentation.
3. Comprehensive Testing (Minimum 6 Tests)
* Structural Tests (Preferred): Verify that code exists, is wired correctly, and adheres to requirements. These should verify file contents rather than execution (e.g., asserting a specific string exists in a file).
   * Target: 70–90% of tests should be structural.
* Behavioral Tests: Execute the code to verify logic. Use these sparingly as they are slower and more fragile.
4. Appropriate Difficulty
* Target Pass Rate: AI agents should pass 1–3 out of 5 runs.
* Too Easy: Agents pass 4–5 times.
* Too Hard/Unclear: Agents pass 0 times.
________________


Test Categories
FAIL_TO_PASS
* Purpose: Tests new functionality or bug fixes.
* Behavior: Fails before the patch is applied; passes after the patch.
* Usage: Most tests will fall into this category.
PASS_TO_PASS
* Purpose: Regression testing.
* Behavior: Passes both before and after the patch.
* Usage: Ensures existing behavior remains intact.

Workflow Guide
Step 1: Initialize Dataset
Initialize your dataset using the Anvil CLI:
Bash
anvil init-dataset \
  --dataset my-dataset \
  --repo-path /path/to/your/repo \
  --base-image golang:1.22


Step 2: Create Task Files
problem.md: Describe the implementation requirements clearly, including edge cases and error handling.
tests.py: Write pytest functions to verify the implementation.
* Example (Structural):
* Python
def test_get_profile_in_interface():
    """Test that GetProfile is defined in the interface."""
    content = Path("/app/my-repo/internal/service/user.go").read_text()
    assert "GetProfile" in content
* * Step 3: Add Task with Diff Capture
Run the following command to capture your solution:
Bash
anvil add-task -d my-dataset \
  --problem-file problem.md \
  --tests-file tests.py \
  --capture-diff


1. Edit: Make the necessary changes to your repository to solve the task.
2. Finalize: Type done when finished.
3. Confirm: Review and accept the captured diff.
4. Reset: The repository will automatically reset for the next task.
Step 4: Validate and Convert
Validate your dataset structure:
Bash
anvil validate-dataset -d my-dataset


Convert to the Anvil format:
Bash
anvil convert-dataset -d my-dataset -u your-username --dockerhub-repo anvil-images


Step 5: Publish and Verify
Publish your Docker images:
Bash
anvil publish-images -d my-dataset -u your-username --repo anvil-images


Oracle Verification (Required):
Verify that your gold patches pass all tests:
Bash
anvil run-evals -d my-dataset --agent oracle -u your-username --dockerhub-repo anvil-images


Agent Evaluation:
Test the difficulty of your task against an AI agent:
Bash
anvil run-evals -d my-dataset \
  --agent mini-swe-agent \
  --model anthropic/claude-sonnet-4-20250514 \
  -u your-username \
  --dockerhub-repo anvil-images

Submission Process
1. Create Submission: Enter your repository details on the platform.
2. Upload Tasks: Upload the task ZIP files containing your problem descriptions, tests, and patches.
3. Validate: Run the system validation for each task.
4. Run Evaluation: Administrators will run agent evaluations (Oracle, Null, and Agent tests).
5. Quality Check: If the agent pass rate is within the target range (1-3/5), a quality check is performed.
6. Submit for Review: Once all 10 tasks are ready, submit the project for human review.
________________


Troubleshooting
Oracle Fails
* Patch Application: Ensure the patch applies cleanly using git apply.
* File Paths: Verify that test paths match the Docker container structure (/app/{repo-name}/...).
* Naming: Ensure fail_to_pass lists match test function names exactly.
Image Build Issues
* Check that Docker is running.
* Verify REGISTRY_USERNAME and REGISTRY_PASSWORD are correct.
* Check Dockerfile syntax.
Agent Pass Rate Issues
* 4-5/5 (Too Easy): Increase complexity by requiring more file changes or adding edge cases.
* 0/5 (Too Hard): Ensure requirements are clear and the oracle passes.

	FAQ
Do I create both the project and tasks?
Yes. You create the repository and all 10 tasks yourself. Tasks should be independent of each other.
What languages are supported?
Python, Go, JavaScript, TypeScript, and others. Choose an appropriate base Docker image.
How complex should tasks be?
Changes across 3+ files, minimum 6 tests. Agent should pass 1-3 out of 5 runs (not 0, not 4-5).
What if agent passes 0/5?
Task may be too hard or have issues. Check that oracle passes first, then review task complexity.
What if agent passes 4-5/5?
Task is too easy. Make it more complex - add edge cases, require more file changes, or make the problem harder.
When can I run quality check?
Only after agent evaluation shows 1-3/5 passes. Quality check is skipped if agent is not in this range.
Can I modify tasks after submission?
Contact us on Slack first. We may need to re-run validation.
